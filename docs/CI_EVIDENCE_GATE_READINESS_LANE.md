# CI Evidence Gate Readiness Lane

Status: readiness lane plus local prototype. No CI enforcement change is made by this document or by the prototype.

LoadBalancerPro is becoming the evidence, governance, and explainability layer for adaptive routing: a local flight simulator and black-box recorder for routing decisions. This lane describes how local Enterprise Lab evidence could later become a CI evidence gate that blocks unsafe adaptive-routing changes.

## Purpose

The future gate would consume local, source-visible Enterprise Lab evidence and fail when a change weakens the proof story or crosses a safety boundary. It is intended to make adaptive-routing changes reviewable before they become release or production-candidate claims.

This lane prepares the gate. It does not create a required check, mutate branch protection, mutate rulesets, change GitHub settings, publish artifacts, sign containers, call cloud resources, or validate real tenants.

## Local Prototype Surface

The local prototype is available from the running app:

- Browser page: `/ci-evidence-gate.html`
- Read-only API: `GET /api/enterprise-lab/ci-evidence-gate-summary`
- Artifact contract: [`CI_EVIDENCE_GATE_ARTIFACT_CONTRACT.md`](CI_EVIDENCE_GATE_ARTIFACT_CONTRACT.md)
- Template/example JSON: [`examples/ci-evidence-gate-summary.template.json`](examples/ci-evidence-gate-summary.template.json)

The endpoint returns deterministic static JSON for local review, including artifact contract metadata such as `artifactVersion`, `artifactKind`, `artifactContract`, and `artifactTemplatePath`. It does not read files, run commands, inspect the live CI environment, call GitHub, call cloud services, read secrets, read environment variables, or mutate filesystem state. The page fetches only the same-origin endpoint and falls back to static prototype copy if the API is unavailable.

The prototype can say `READY_FOR_LOCAL_REVIEW`, `NOT_ENFORCED`, `PASS_STYLE`, `WARN_STYLE`, and `FAIL_STYLE_BLOCKER` as review semantics. Those labels are local readiness semantics only. They are not live merge enforcement, not required checks, not branch protection, and not a production certification result.

The template/example JSON is not a generated run result. It is a source-visible packet shape for reviewer handoff and future parser design.

## Future Pass Conditions

A future CI evidence gate could pass only when:

- Enterprise Lab workflow evidence exists and labels the run as lab evidence only;
- controlled policy evidence covers `off`, `shadow`, `recommend`, and guarded `active-experiment` modes;
- observability evidence contains process-local metrics only and does not require live Prometheus, Grafana, cloud, or secrets;
- performance evidence uses local loopback fixtures and warning-only thresholds, without fixed benchmark claims;
- enterprise auth proof uses synthetic mock IdP/JWKS fixtures and no real tenant material;
- adaptive-routing experiment evidence keeps default behavior unchanged unless an explicit guarded mode is selected;
- evidence files contain no secrets, real tokens, tenant IDs, private endpoints, or customer data;
- evidence paths stay under ignored `target/` directories;
- no reviewed change adds release, registry, signing, cloud mutation, private-network validation, or GitHub settings mutation instructions.

## Future Fail Conditions

A future gate should fail or block review when:

- required local evidence is missing, malformed, or produced for the wrong commit;
- evidence claims production certification, production performance, production SLO/SLA proof, live-cloud validation, real-tenant validation, real enterprise IdP validation, signed-container proof, registry publish completion, or governance-applied proof;
- adaptive-routing changes remove guardrails, hide rollback/fail-closed reasons, or make active-experiment behavior implicit;
- generated evidence is committed outside ignored `target/` paths;
- evidence contains secrets, tenant-specific identifiers, private endpoints, customer data, bearer tokens, API keys, or credentials;
- commands require release creation, registry login or publish, container signing, cloud mutation, private-network validation, GitHub settings mutation, or branch deletion;
- performance evidence presents fake benchmark numbers or fixed production thresholds as proof.

## What Remains Manual Today

Reviewers still manually confirm:

- latest CI and CodeQL passed for the exact commit under review;
- the local prototype at `/ci-evidence-gate.html` and `GET /api/enterprise-lab/ci-evidence-gate-summary` still reports `NOT_ENFORCED`;
- Dependency Review and Trivy findings are resolved or have owner-approved rationale;
- the generated local evidence, if produced, is from the same commit;
- `release-downloads/` was not changed;
- no generated `target/` evidence was added to the PR;
- no GitHub settings, branch protection, rulesets, secrets, environments, required checks, releases, tags, registry images, or container signatures were changed.

## Intentionally Not Enforced Yet

This lane does not add or change `.github/workflows/` files. It does not add required checks. It does not enforce branch protection, change rulesets, mutate environments, or change repository settings.

It also does not prove production readiness, production performance, production SLO/SLA behavior, live-cloud behavior, real tenant behavior, real enterprise IdP behavior, signed-container provenance, registry publication, or governance-applied status.

Use this lane, the prototype, and [`CI_EVIDENCE_GATE_ARTIFACT_CONTRACT.md`](CI_EVIDENCE_GATE_ARTIFACT_CONTRACT.md) as the design and safety contract for a later implementation sprint. A future implementation can add a local parser or CI job only after the artifact contract, evidence schema, failure policy, runtime cost, and reviewer workflow are explicitly approved.
