Signal Interpretation Guide

Reviewer-facing interpretation for visible local lab signals.

Interpret these as visible lab signals derived from the local comparison response and request payload. Exact production scoring is not claimed unless exposed by the API.

Healthy vs unhealthy backend state

A healthy backend can be selected; an unhealthy backend should be treated as degraded lab input and inspected before trusting any selection.

Lower or higher latency

Lower visible latency usually supports selection under latency-aware strategies, while higher p95 latency is a pressure signal for reviewer interpretation.

Load or active connection pressure

Higher in-flight request counts or queue depth indicate local lab pressure that may explain a backend change under the same strategy.

Capacity/weight assumptions where exposed

Capacity, estimated concurrency, and weight help explain local preference when present; the cockpit does not invent hidden production scoring.

Unchanged backend with changed signals

If the backend is unchanged but visible lab signals changed, compare reason text and deltas before calling the result equivalent.

Backend changes under the same strategy

If the strategy is unchanged but the backend changed, visible health, latency, load, queue, capacity, or weight shifts may have changed the preferred candidate.

Strategy changes but backend remains the same

If the strategy changed but the backend stayed the same, multiple strategies can still prefer the same controlled lab candidate.

All backends degraded or unhealthy

All-unhealthy lab input is a degradation boundary and should be reviewed as a failure/fallback scenario, not production proof.

Missing or unavailable local API data

If local API data is missing or unavailable, static guidance and evidence links remain available, but the monitored decision chain is not complete.

How This Routing Decision Was Made

Reviewer/operator answers for controlled lab routing-proof questions.
How was the selected backend chosen?

The first returned comparison result is the primary controlled lab decision. It names the selected backend/server, includes returned reason text, and should be read against the visible candidate list.

How did the selected strategy influence the decision?

The selected strategy applies its routing behavior to visible local lab candidates; exact production scoring is not claimed unless exposed by the API.

How did visible input signals influence the proof?

The cockpit explains visible input signals from the local lab response: backend health, latency, load, active connections, capacity, weight, error rate, queue depth, and network-awareness fields when present.

How should reviewers interpret close candidates?

If multiple candidates appear close, compare returned reason text and visible signals; the cockpit explains observable lab evidence and does not invent hidden scoring.

How should unchanged backend with changed signals be read?

The selected backend is unchanged, but visible lab signals changed; review the strategy reason and signal deltas before treating the result as equivalent.

How should changed backend with same strategy be read?

Backend changed but strategy stayed the same: visible health, load, latency, queue, capacity, or weight shifts can change which candidate the same strategy prefers.

How should changed strategy with same backend be read?

Strategy changed but backend stayed the same: multiple strategies can still prefer the same controlled lab candidate.

How are unhealthy/degraded backends represented?

Unhealthy backends are counted from healthy=false; all-unhealthy payloads are treated as a degradation boundary, not production proof.

How does recovery change the evidence story?

Recovery is represented when visible lab inputs show fewer unhealthy candidates or reduced pressure compared with the packaged baseline; it remains controlled lab evidence.

What can this controlled lab run prove?

It can prove local reproducibility, same-origin compare API behavior, visible routing explanation, local scenario deltas, evidence association path, and browser-local reviewer handoff text.

What can it not prove for production?

It does not prove production traffic, production telemetry, live cloud behavior, real tenant behavior, SLA/SLO, registry publication, container signing, or production certification.

How to Read Lab Edge Cases

  • Page loaded before interaction: static reviewer guidance remains available before a lab run.
  • No previous lab scenario has been captured yet.
  • Same lab scenario run twice or unchanged from baseline: strategy order and input signal totals should read as unchanged.
  • The selected backend is unchanged, but visible lab signals changed.
  • Backend changed but strategy stayed the same: visible lab signal deltas likely changed which candidate the same strategy preferred.
  • Strategy changed but backend stayed the same: the selected backend can remain preferred under multiple routing methods.
  • All backends unhealthy/degraded: every visible backend is unhealthy, so treat the result as a degradation boundary instead of production proof.
  • Empty comparison response: The local lab API returned an empty comparison response; static reviewer guidance remains available.
