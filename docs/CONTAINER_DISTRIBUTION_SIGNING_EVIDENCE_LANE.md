# Container Distribution Signing Evidence Lane

## Verdict

This is a **future gated** lane for container distribution and signing evidence.

- No container publishing is performed by this sprint.
- No container signing is performed by this sprint.
- No release, tag, or GitHub Release is created by this sprint.
- This does not make LoadBalancerPro production certified or enterprise-production ready.

The lane is documentation and guardrail work only. It prepares reviewer-visible evidence expectations for a later approved implementation sprint.

## Purpose

This lane prepares future reviewer-visible distribution trust evidence for:

- image naming and versioning policy;
- registry decision;
- signing approach;
- key management decision;
- SBOM and provenance expectations;
- verification commands;
- approval gates.

## Current State

The current repository supports only bounded local and CI container evidence:

- A checked-in multi-stage [`../Dockerfile`](../Dockerfile) builds the Spring Boot JAR and runs it as the non-root `loadbalancer` user.
- The Dockerfile defaults to `SPRING_PROFILES_ACTIVE=prod` and includes a container-local healthcheck.
- [`CONTAINER_DEPLOYMENT.md`](CONTAINER_DEPLOYMENT.md) documents local-only Docker build/run recipes, loopback binding, API-key boundary checks, and no-registry-publish boundaries.
- CI builds the Docker image, runs loopback-bound container health smoke, and runs Trivy for fixed high/critical findings.
- CI generates CycloneDX SBOM artifacts for the Maven/JAR dependency graph.
- The tag-triggered Release Artifacts workflow builds deterministic JAR/SBOM/checksum assets and GitHub artifact attestations for approved semantic tags.
- [`CONTAINER_SIGNING_DECISION_RECORD.md`](CONTAINER_SIGNING_DECISION_RECORD.md) and [`CONTAINER_REGISTRY_SIGNING_ROLLOUT.md`](CONTAINER_REGISTRY_SIGNING_ROLLOUT.md) keep registry publication and signing deferred.

Not yet implemented:

- registry publication;
- canonical release image naming;
- release image tags;
- image signing;
- container-image attestation publication;
- image-level SBOM attachment;
- rollback and retention evidence for published image digests.

## Future Verification Commands

Example future commands only. They are not run in this sprint and must not be used until the required decisions and approval gates are satisfied.

```bash
docker build -t ghcr.io/RicheyWorks/loadbalancerpro:${VERSION} .
trivy image --severity HIGH,CRITICAL --ignore-unfixed ghcr.io/RicheyWorks/loadbalancerpro:${VERSION}
syft ghcr.io/RicheyWorks/loadbalancerpro:${VERSION} -o cyclonedx-json
cosign sign --yes ghcr.io/RicheyWorks/loadbalancerpro:${VERSION}
cosign verify ghcr.io/RicheyWorks/loadbalancerpro:${VERSION}
cosign attest --predicate image-sbom.json ghcr.io/RicheyWorks/loadbalancerpro:${VERSION}
cosign verify-attestation ghcr.io/RicheyWorks/loadbalancerpro:${VERSION}
```
